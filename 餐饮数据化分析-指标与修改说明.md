# 餐饮数据化分析 — 完整说明文档

本文档面向**业务人员**与**后续开发阅读**，说明当前项目的**文件结构**、**数据源与字段**、**宽表生成逻辑**、以及**分析结论表（12 个 Sheet）各自使用的底表与生成逻辑、指标标准**。便于完全掌握数据结构与开发结构。

---

## 一、项目与文件结构

### 1.1 根目录（运行分析脚本的目录）

| 类型 | 路径/文件 | 说明 |
|------|------------|------|
| **主入口（推荐执行的命令）** | `main_analysis.py` | 多门店调度器。检测 `data/` 下的门店子目录，在每个门店目录中调用一次 `main_analysis_v2_backup.py`，并整理输出到 `output/<门店>/<数据段>/`，生成 `output/index.json`。 |
| **核心分析脚本** | `main_analysis_v2_backup.py` | 单门店分析逻辑：读取当前工作目录下的 Excel 源数据、生成宽表、生成分析结论 Excel、生成单个 `report.json`。通常不直接在根目录手动运行，而是由 `main_analysis.py` 调用。 |
| **使用说明** | `使用说明.md` | 运行命令、前端访问方式（端口 8080 / 5173） |
| **本说明** | `餐饮数据化分析-指标与修改说明.md` | 本文档 |
| **单页前端** | `index.html`、`styles.css` | 老版报告页，依赖 `report.json`，需在根目录起 HTTP 服务（如 8080） |
| **备份/旧版** | `备份/`、`old/` | 历史版本脚本，非当前主流程 |

### 1.2 数据目录（按门店）

- 多门店模式下，每个门店对应一个**数据目录**（如 `data/紫竹餐厅`、`data/夕佳悦餐厅`）。
- 运行分析时，当前工作目录会切换到该门店目录，因此**所有 Excel 源文件路径均相对于当前工作目录**。
- 必选文件需放在当前工作目录下：`支付明细.xlsx`、`销售明细.xlsx`、`退菜明细.xlsx`、`订单明细.xlsx`、`优惠明细.xlsx`。
- 可选文件（见下节）若存在则参与分析，不存在则对应分析为空表，不报错。

### 1.3 输出目录

- 路径：`output/<餐厅名>/<数据起止日期>/`
  - 餐厅名：来自当前工作目录的目录名（如 `os.getcwd()` 的 basename）。
  - 数据起止日期：由支付明细的 `time` 最小/最大日期得到，格式 `YYYY-MM-DD_YYYY-MM-DD`。
- 该目录下生成：
  - **宽表**：`餐饮业务全量宽表_<餐厅名>_<日期范围>_<时间戳>.xlsx`
  - **分析结论**：`餐饮数据化分析结论_<餐厅名>_<日期范围>_<时间戳>.xlsx`（多 Sheet）
  - **前端数据**：`report.json`（与结论表内容一致的 JSON 摘要）

### 1.4 前端（React 仪表盘）

- 目录：`dashboard/`
- 关键文件：
  - `dashboard/src/App.tsx`：主布局，读取 `public/report.json`
  - `dashboard/src/components/`：`ModuleSales.tsx`、`ModuleUser.tsx`、`ModuleDish.tsx`、`ModuleStaffLost.tsx`、`ModuleRemarks.tsx`、`Header.tsx`、`DataTable.tsx`、`AnimatedNumber.tsx`
  - `dashboard/src/data/report.ts`：报告数据类型定义
- 运行：`cd dashboard && npm install && npm run dev`（开发）或 `npm run build && npm run preview`（构建后预览）。
- 报告数据需将 `report.json` 放到 `dashboard/public/report.json`，与 `output/.../report.json` 内容一致。

## 二、数据源：必选表与可选表

### 2.1 必选表（FIELD_CONFIG）

脚本通过「关键词模糊匹配」将 Excel 列名映射为**内部字段名**。下表列出各表文件名与内部字段（业务理解以内部字段为准）。

| 配置键 | 文件名 | 内部字段（mapping） | 说明 |
|--------|--------|---------------------|------|
| **pay** | 支付明细.xlsx | id, amount, payer, time, method, staff | 一行=一笔支付；**仅用于可识别用户相关分析**，不用于销售趋势/时段交叉的金额与订单数 |
| **sales** | 销售明细.xlsx | id, dish, qty, price, staff, remark, type | 一行=一条菜品销售 |
| **refunds** | 退菜明细.xlsx | id, dish, qty, price, staff, remark, time | 一行=一条退菜 |
| **orders** | 订单明细.xlsx | id, people, status, time, refund_tag, table, order_remark, discount_method, sensitive_action | 一行=一张订单；id 会经 clean_id 清洗 |
| **discounts** | 优惠明细.xlsx | id, discount_amount, discount_name, discount_reason, discount_type | 一行=一种优惠在某单上的一条记录 |

- **id** 列：若原始列名为「订单号」「订单编号」「业务单号」「单号」等，会统一为 `id`，并做 `clean_id`（去空、0、nan、.0 等）。
- **time**：订单、支付、退菜三表会统一转为 `pd.to_datetime`。

## 三、宽表生成逻辑与底表、字段

### 3.1 宽表的业务含义与粒度

- **一行**：表示「某订单下的某一条菜品行（销售或退菜）」+ 该订单的支付、优惠等信息。
- **主键逻辑**：订单 id + 菜品行（同一订单多条菜品会变成多行）。

### 3.2 生成步骤（代码顺序）

1. **合并销售与退菜为“菜品流水”**
   - 在销售明细上增加列 `dish_status = "SALE"`，在退菜明细上增加 `dish_status = "REFUND"`。
   - `df_all_dishes = pd.concat([df_sales, df_refunds], ignore_index=True)`。
   - 带来的字段：id, dish, qty, price, staff, remark, type, dish_status（以及退菜表的 time，若存在会与订单/支付 time 并存）。

2. **订单级优惠汇总**
   - `df_discounts_agg`：按 `id` 聚合优惠明细，`discount_amount` 求和，`discount_name` 去重后用 `"; "` 拼接。
   - 字段：id, discount_amount, discount_name。

3. **以订单明细为左表，依次左连接**
   - 只保留 `id != ""` 的订单。
   - **第一 merge**：订单 × df_all_dishes，on=id，how=left → 得到「订单 + 菜品行」粒度，带订单所有列 + 菜品列（dish, qty, price, staff, remark, type, dish_status）。
   - **第二 merge**：上表 × df_pay，on=id，how=left，suffixes=("", "_pay") → 带入支付列（amount, payer, time 或 time_pay, method, staff 等）；若同一订单多笔支付会出现多行。
   - **第三 merge**：上表 × df_discounts_agg，on=id，how=left → 带入 discount_amount, discount_name。
   - **第四 merge（若订单表有）**：再与订单表的 `[id, discount_method, sensitive_action]` 按 id left 一次，保证宽表上显式有优惠方式与敏感操作（避免被前序 merge 覆盖）。

4. **输出**
   - 保存为 `餐饮业务全量宽表_<餐厅名>_<日期范围>_<时间戳>.xlsx`，路径见 1.3。

### 3.3 宽表主要字段来源汇总

| 来源 | 主要字段 |
|------|----------|
| 订单明细 | id, people, status, time, table, order_remark, discount_method, sensitive_action, refund_tag 等 |
| 销售/退菜 | dish, qty, price, staff, remark, type, dish_status |
| 支付明细 | amount, payer, time（或 time_pay）, method, staff（可能带 _pay 后缀） |
| 优惠汇总 | discount_amount, discount_name |

- 若订单表或宽表后续被读取时带有「支付合计」「订单优惠」「菜品优惠」「营业日期」等列名（来自导出系统），也会在 7_优惠分析 等环节通过 `find_actual_column` 使用，但宽表**标准生成过程**不强制要求这些列存在。

### 3.4 宽表字段 → 指标字典（关键指标溯源）

下表列出在后续分析中频繁使用的核心指标，在宽表中对应的字段及其来源底表，便于检查口径与追溯：

| 指标/字段（宽表列） | 业务含义 | 来源底表 | 底表字段/计算方式 |
|----------------------|----------|----------|--------------------|
| `id` | 订单唯一标识 | 订单明细 | 映射自「订单号/订单编号/单号/业务单号」等，经 `clean_id` 清洗 |
| `people` | 用餐人数 | 订单明细 | 用餐人数/人数 |
| `time` | 订单时间（下单/开台时间） | 订单明细 | 订单时间列，统一转为 datetime |
| `status` | 订单状态 | 订单明细 | 订单状态/状态 |
| `order_remark` | 整单备注 | 订单明细 | 备注/订单备注等 |
| `discount_method` | 优惠方式（平台券/团购/手工折扣等） | 订单明细 | 优惠方式/折扣方式等，原样带入宽表 |
| `sensitive_action` | 敏感操作标签（如订单手动打折） | 订单明细 | 敏感操作/风控操作等，原样带入宽表 |
| `dish` | 菜品名称 | 销售/退菜明细 | 菜品名称/项目名称 |
| `qty` | 菜品数量 | 销售/退菜明细 | 销售数量/数量 |
| `price` | 行级金额（销售额或退菜额） | 销售/退菜明细 | 金额/销售额；退菜时通常为负值或正值视系统导出而定 |
| `dish_status` | 行状态：SALE/REFUND | 派生 | 销售明细打 `SALE`，退菜明细打 `REFUND` 后合并 |
| `amount` | 支付金额 | 支付明细 | 交易金额/金额 |
| `payer` | 付款人标识 | 支付明细 | 付款人/客户/会员号等 |
| `method` | 支付方式 | 支付明细 | 支付方式/结算方式 |
| `time_pay`/`time`（支付） | 支付时间 | 支付明细 | 交易时间/支付时间等 |
| `discount_amount` | 订单级总优惠金额 | 优惠明细聚合 | 对同一 id 的优惠行 `discount_amount` 求和 |
| `discount_name` | 订单级优惠名称集合 | 优惠明细聚合 | 同一 id 下所有优惠名称去重后用 `"; "` 拼接 |
| `discount_reason` | 订单级优惠原因集合 | 优惠明细聚合 | 同一 id 下 `discount_reason` 去重拼接 |
| `discount_type` | 订单级优惠类型集合 | 优惠明细聚合 | 同一 id 下 `discount_type` 去重拼接 |
| 「订单优惠(元)」 | 订单口径优惠金额 | 宽表/导出系统 | 若存在此列，则被 7_优惠分析 用作订单级优惠金额；否则回退到 `discount_amount` |
| 「支付合计(元)」 | 订单口径支付合计 | 订单明细/导出系统 | 若存在，被用来构造 `order_amount`；否则从销售明细按订单汇总 `price` |
| 「营业日期」 | 业务日期（去掉时间） | 订单/支付/宽表 | 若存在对应列，会用作日期维度；否则由 `time.dt.normalize()` 计算 |

> 注：上表为关键字段速查，并非宽表所有列的穷举。其它辅助字段（桌号、用餐方式、菜品分类等）在具体分析模块中按需引用。

## 四、营业时段划分（全表统一）

所有按“时段”的统计均使用 `classify_daypart(time)`：

| 时段 | 小时范围（含左不含右） |
|------|------------------------|
| 早餐 | 6:00–10:59 |
| 午餐 | 11:00–14:59 |
| 下午茶 | 15:00–17:59 |
| 晚餐 | 18:00–21:59 |
| 夜宵 | 22:00–5:59 |

- 时间为 NaT 或解析失败时返回空字符串；2_时段交叉 会过滤掉「星期」或「时段」为空的记录。

## 五、销售金额与订单数的统一口径（重要）

- **销售金额**（总销售额、月/周趋势、时段交叉中的销售额、KPI 中的总/日均销售额）  
  - **底表**：**全部订单**（订单明细）。  
  - **金额取值**：优先订单表中的「支付合计(元)」；若无该列则用「销售明细按订单 id 汇总的 price 之和」作为 `order_amount`。  
  - **目的**：避免仅用支付明细导致漏掉无付款人标识的支付。

- **订单数**（总客流量、月/周订单数、时段交叉中的订单数、KPI 中的总/日均订单量）  
  - **底表**：仅 **status == "已结账"** 的订单。  
  - **统计**：按订单 id 去重计数。

- **支付明细**  
  - 仅用于：用户分层（RFM）、各层级时段/支付方式/优惠响应、高退菜客户、菜品主攻客群、关键备注词时间等与「可识别用户」相关的分析。

## 六、分析结论表（12 个 Sheet）底表与生成逻辑

### 6.1 1_销售趋势

| 输出块 | 底表 | 生成逻辑 | 指标/标准 |
|--------|------|----------|------------|
| 核心结论（3 行文字） | total_sales, total_orders, aov, total_people, daily_*, lost_* | 由下方 KPI 与流失汇总拼接 | — |
| 核心指标矩阵 | df_orders_amt, df_orders_paid, 流失订单 | 总销售额=全量 order_amount.sum()；总客流量=已结账 id.nunique()；AOV=总销售额/总客流量；总用餐人数=已结账订单 people 之和；日均=总量/跨度天数；流失订单数=退款完成+已撤单 | 见第五节 |
| 流失订单汇总 | refund_orders, cancel_orders | 按状态汇总订单数 | 流失=退款完成+已撤单 |
| 月度趋势与 MoM | df_orders_amt（金额）, df_orders_paid（订单数） | 按 month 聚合：销售额=order_amount.sum()，订单数=id.nunique()；环比=pct_change() | 金额全量、订单数已结账 |
| 周度趋势与 WoW | 同上 | 按 week 聚合；仅保留完整自然周（起止在数据范围内）；环比=pct_change() | 同上 |

### 6.2 2_时段交叉

| 输出块 | 底表 | 生成逻辑 | 指标/标准 |
|--------|------|----------|------------|
| 星期×时段(销售额与占比) | df_orders_amt（金额）, df_orders_paid（订单数） | 按订单 time 派生 weekday_cn、daypart；金额按 order_amount.sum()，订单数按 id.nunique()；合并后算营收占比；**过滤掉星期或时段为空的记录** | 只保留有明确星期和时段的行 |
| 最忙 Top3 与冷清 Bottom | 同上 time_pivot | 按销售额排序取前 3、后 3 | — |

### 6.3 3_退菜分析

| 输出块 | 底表 | 生成逻辑 | 指标/标准 |
|--------|------|----------|------------|
| 退菜率 Top5 菜品 | df_refunds, df_sales | 按菜品汇总退菜 qty/price/id；退菜率=退菜金额/该菜在销售明细中的销售额；**无正向销售仅有退菜时退菜率显示「无正向销售，仅退菜」**；再按退菜金额取前 5 | 选出=按退菜金额 Top5；退菜率口径见上 |
| 退菜备注高频词 | df_refunds | remark 去空后 value_counts | — |
| 高退菜客户 | df_refunds, df_pay | 退菜订单按 payer 统计订单数；支付明细按 payer 统计总订单数；合并后保留「总订单数≥2 且 退菜订单数>0」，算退菜订单占比 | 总订单数/退菜订单数均来自支付明细 |

### 6.4 4_用户分层

| 输出块 | 底表 | 生成逻辑 | 指标/标准 |
|--------|------|----------|------------|
| 用户分层汇总 | df_pay（payer 非空） | RFM：last_visit, frequency, monetary, R_days；分层规则：frequency==1→新客；≥2 且 R>65→沉睡唤醒；≥2 且 R≤65 且 monetary≥100→高价值忠诚；其余≥2→一般活跃；再按分层统计人数、平均消费、平均频次、平均R天、订单数 | 见上文分层规则 |
| 各层级响应概览 | segment_summary, 高退菜客户, 各层级优惠响应(有折扣用户), 优惠用户特征 | 以 `segment_summary` 为基准行：1）将 3_退菜分析的「高退菜客户」按付款人 × `user_rfm.segment` 汇总为「高退菜客户数/高退菜订单数」并按 segment 合并；2）将 7_优惠分析的「优惠用户特征(谁/几次/金额)」按用户类型汇总为「平台优惠客户数/平台优惠订单数/平台优惠金额」并按 segment 合并；3）将本 Sheet 的【各层级优惠响应(有折扣用户)】按 segment 合并为「有折扣用户数/有折扣用户订单数/有折扣用户金额/总订单数/有折扣订单占比」。最终形成按分层的“三响应 + 高退菜 + 基础 RFM 指标」总表。 | 三类人群口径：高退菜客户=高退菜客户表；平台优惠客户=7_优惠分析中“有优惠方式+无敏感操作”的用户；有折扣用户=宽表中 `discount_name` 有值且 `discount_amount>0` 的用户 |
| 各层级优惠响应(有折扣用户) | 宽表 df_wide, user_rfm | 从宽表中筛选：`discount_name` 字符串非空且≠"--"、且 `discount_amount>0` 的订单；按 id 去重后，合并到 `user_rfm` 得到每笔订单所属分层；再按 segment 汇总：有折扣用户数=付款人去重数，有折扣用户订单数=订单 id 去重数，有折扣用户金额=discount_amount 之和，总订单数=该分层订单总数；有折扣订单占比=有折扣用户订单数/总订单数。 | 「有折扣用户」完全基于宽表 `discount_name+discount_amount` 口径，与 7_优惠分析中的「优惠用户特征」区分开 |
| 分层用户明细 | user_rfm | 付款人、最近消费时间、订单数、总消费金额、用户类型 | — |

### 6.5 5_员工表现

## 八、前端 report.json 与仪表盘

- **report.json**：与分析结论 Excel 同一次 run 生成，位于 `output/.../report.json`。结构概览：
  - `meta`：generatedAt, dataRange, restaurant, rangeKey, wideFile, reportFile
  - `overview`：summary（核心结论）, cards（总销售额、已结账订单、平均客单价、用户分层人数）
  - `sections`：与 1～12 个 Sheet 对应，每项含 id、title、conclusions、tables（name + columns + rows）；占比类在写入前格式化为 xx.xx% 字符串。

- **dashboard**：读取 `public/report.json`，用各 Module 组件展示对应 section 的 tables 与 conclusions；不改变任何业务口径。
