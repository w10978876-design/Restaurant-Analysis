## 二、第一步：前端数据加载（餐厅 + 周期切换）

目标：页面加载时从 `output/index.json` 拉取餐厅与周期列表，Header 下拉选择后按「餐厅 + rangeKey」请求对应 `report.json`，整体仍是一个单页，不拆成多个 dashboard。

### 2.1 约定与前提

- 前端访问方式：在**项目根目录**起 HTTP 服务（如 `python -m http.server 8080`），根路径 = 项目根，故：
  - 索引：`/output/index.json`
  - 某周期报表：`/output/夕佳悦餐厅/2026-02-01_2026-02-28/report.json`  
  即 `reportPath` 在 index 中若为 `output/.../report.json`，前端 fetch 时需加首斜杠：`/${period.reportPath}`。
- 不要求改后端；仅改 `dashboard/` 内代码，构建后把 `dashboard/dist/` 内容放到根目录或通过同一 8080 提供（见 2.5）。

### 2.2 数据类型（`dashboard/src/data/`）

- **新增** `index.ts`（或并入 `report.ts`）：
  - 定义 `IndexPeriod`：`rangeKey`, `dataRange`, `reportPath`（及可选 `wideFile`, `reportFile`）。
  - 定义 `IndexRestaurant`：`id`, `name`, `periods: IndexPeriod[]`。
  - 定义 `ReportIndex`：`restaurants: IndexRestaurant[]`（与当前 `output/index.json` 结构一致）。
- **可选**：在 `report.ts` 的 `ReportMeta` 中已有 `restaurant`、`rangeKey` 等，与后端 meta 保持一致即可。

### 2.3 加载索引与报表

- **加载索引**：
  - 新增 `loadIndex(): Promise<ReportIndex>`，请求 `GET /output/index.json`。
  - 若 404 或失败：可视为「未跑过多门店分析」，前端展示友好提示（如「请先运行分析脚本并生成 output/index.json」）。
- **加载报表**：
  - 修改 `loadReport` 的语义为「按当前选中的餐厅 + 周期加载」：
    - 签名可为 `loadReport(reportPath: string): Promise<Report>`，其中 `reportPath` 来自当前选中 period（如 `output/夕佳悦餐厅/2026-02-01_2026-02-28/report.json`）。
    - 请求 URL：`/${reportPath}`（避免重复斜杠）。
  - 取消「单一全局 cachedReport」或改为按 reportPath 缓存，避免切换周期后仍显示旧数据。

### 2.4 App 与 Header 状态与联动

- **App 状态建议**：
  - `reportIndex: ReportIndex | null`（索引）
  - `selectedRestaurantId: string | null`（当前选中餐厅 id）
  - `selectedRangeKey: string | null`（当前选中周期 rangeKey）
  - `report: Report | null`（当前报表）
  - `loading / error`（加载中/失败）

- **初始化（仅一次）**：
  - 请求 `loadIndex()` → 写入 `reportIndex`。
  - 若 `restaurants.length > 0`，取第一个餐厅、其第一个 period，设 `selectedRestaurantId`、`selectedRangeKey`，再调用 `loadReport(period.reportPath)` → 写入 `report`。
  - 若无餐厅：展示空态（见 2.6）。

- **Header 扩展**：
  - 「餐厅选择」：用 `reportIndex.restaurants` 渲染 `<select>`，value = `restaurant.id`，显示 `restaurant.name`。
  - 「报告周期」：用当前选中餐厅的 `periods` 渲染 `<select>`，value = `period.rangeKey`，显示 `period.dataRange` 或简短文案（如「2026-02-01 至 2026-02-28」）。
  - 选择变化时：`onRestaurantChange` / `onPeriodChange` 更新 `selectedRestaurantId` / `selectedRangeKey`，并依据新选中项取 `reportPath`，再 `loadReport(reportPath)` → 更新 `report`（并 setLoading(true) → 请求完成后 setReport / setError、setLoading(false)）。

- **标题/副标题**：可用 `report?.meta?.restaurant` 或 `selectedRestaurantId` 显示当前餐厅名；周期可用 `report?.meta?.dataRange` 或当前 period 的 `dataRange`。

### 2.5 开发与生产环境下的静态资源路径

- **开发**（`npm run dev`）：Vite 默认以项目根为 dev server 根。若前端在 `dashboard/`，需让请求 `/output/*` 落到项目根下的 `output/`：
  - 在 `vite.config.ts` 中配置 dev server 的 proxy：将 `/output` 代理到同一 host 的根目录下的 `output`（或代理到 8080 上挂载的静态目录），这样 `fetch('/output/index.json')` 能拿到真实文件。
  - 或者：开发时用「根目录起 8080 + 把 dashboard 构建产物放到根目录」的方式，则无需 proxy，直接请求 `/output/index.json` 即可。
- **生产**（日常使用）：按你当前习惯，在根目录 `python -m http.server 8080`，把 `dashboard/dist` 内容复制到根目录（或 8080 的 DocumentRoot 指向包含 `index.html` 与 `output/` 的目录），保证 `/output/index.json` 与 `/output/<餐厅>/<rangeKey>/report.json` 可访问。

### 2.6 空态与错误

- **无索引或 restaurants 为空**：提示「请先运行分析脚本生成 output/index.json」，并说明运行 `main_analysis.py` 等。
- **某 report.json 加载失败**：保留当前选中餐厅/周期，仅清空或保留上一份 report，并提示「该周期报表加载失败，请检查 output 目录或重新运行分析」。

## 三、第二步：分析内容增加后的前端展示

目标：后端 report.json 已包含 12 个 section（1_sales～12_visitor），前端目前只展示了其中一部分；需补齐展示，使「分析内容已经增加」在页面上可见。

### 3.1 策略选择

- **方案 A（推荐）**：**通用 Section 组件**  
  新增一个 `ModuleSection`（或 `GenericSection`），接收 `report` 和 `sectionId`，内部用 `findSection(report, sectionId)` 取数据，统一渲染：  
  - `title` 作为卡片标题；  
  - `conclusions` 列表；  
  - 各 `tables` 用现有 `DataTable` 或简单表格展示。  
  然后在 App 中按 section 顺序渲染 12 个模块（1_sales～12_visitor），其中 1/4/5/6/8 可继续用现有精细化组件（ModuleSales、ModuleUser、ModuleStaffLost、ModuleDish、ModuleRemarks），其余 2/3/7/9/10/11/12 用 `ModuleSection` 展示，避免重复造轮子且易于维护。

- **方案 B**：为 2/3/7/9/10/11/12 各写一个专用模块（如 ModuleTimeCross、ModuleRefund、ModuleDiscount…）。  
  展示效果可更贴合业务，但工作量大；可放在后续迭代，先由通用 Section 覆盖。

### 3.2 推荐实施（方案 A）

1. **新增** `dashboard/src/components/ModuleSection.tsx`：
   - Props：`report: Report | null`，`sectionId: string`（如 `'2_time'`）。
   - 使用 `findSection(report, sectionId)`；若 section 为空则 return null。
   - 布局：与现有模块风格一致（圆角卡片、标题、结论列表、表格列表）；表格用 `DataTable` 或现有表格组件。
   - 若某 section 无 tables 仅 conclusions，仅渲染结论即可。

2. **App 中模块顺序**（在「数据加载」改造完成后）：
   - 保持：OverviewBlock、ModuleSales、ModuleUser、ModuleDish、ModuleStaffLost、ModuleRemarks（这些内部已绑定 1_sales、4_user、5_staff、6_dish、8_lost）。
   - 新增：在合适位置插入 `ModuleSection` 对应 2_time、3_refund、7_discount、9_sensitive、10_groupon、11_soldout、12_visitor（可按 1～12 顺序统一排，或把通用 Section 放在现有模块之后，视 UI 而定）。

3. **避免重复**：  
   - 若 1_sales 既在 OverviewBlock/ModuleSales 中展示，又用 ModuleSection(report, '1_sales')，会重复。  
   - 因此只对「当前没有专用模块」的 section 使用 ModuleSection：2、3、7、9、10、11、12。

### 3.3 可选优化（后续）

- 空 section：若某周期无团购/无沽清/无进馆数据，后端可能对应 section 的 tables 为空；ModuleSection 可对「conclusions 与 tables 皆空」时折叠或显示「本周期无数据」。
- 表格列宽、导出等：若需与现有 DataTable 一致，可复用同一套样式与逻辑。

## 四、实施顺序与检查点

| 步骤 | 内容 | 完成标准 |
|------|------|----------|
| 1 | 数据类型：Index + 可选 ReportMeta 扩展 | `loadIndex()` 能解析当前 `output/index.json`；`loadReport(path)` 能按 path 请求并解析 report.json |
| 2 | App 状态与初次加载 | 进入页面 → 请求 index → 选第一个餐厅+周期 → 请求对应 report.json → 当前报表正常显示 |
| 3 | Header 下拉与切换 | 餐厅/周期下拉选项来自 index；切换后重新请求 report 并刷新下方内容 |
| 4 | 开发/生产路径 | 开发时能访问 `/output/index.json` 与各 report（proxy 或统一 8080）；生产 8080 根目录服务可访问同样路径 |
| 5 | 空态与错误提示 | 无 index、无餐厅、report 加载失败时有明确提示 |
| 6 | ModuleSection 组件 | 能根据 sectionId 渲染任意 section 的 title、conclusions、tables |
| 7 | 补齐 2/3/7/9/10/11/12 展示 | 页面上能看到时段交叉、退菜、优惠、敏感操作、团购、沽清、进馆转化等内容（即使用通用表格展示） |

## 五、与「两步使用习惯」的对应

- **第一步**：运行一次 `python main_analysis.py`（或你当前入口），生成多餐厅多周期 output 及 `output/index.json`。
- **第二步**：在根目录 `python -m http.server 8080`（或 `./启动前端服务.sh`），打开 http://127.0.0.1:8080，在页面内通过「餐厅 + 报告周期」下拉切换，无需多个 dashboard、无需改 URL。

只有在你需要改 React 源码时，才进入 `dashboard/` 做 `npm run build`，并将 dist 拷贝到根目录（或配置 8080 指向含 output 与前端静态资源的目录）。
